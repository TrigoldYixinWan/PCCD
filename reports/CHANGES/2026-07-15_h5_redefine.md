# Change: Replace H5 with a data-backed hard-safety dimension (severity: RED — needs sign-off)

Date / commit: 2026-07-15 / (opened by PaperGuru; awaiting PKU category data before finalizing)

Trigger: Day-2 static audit (reports/day2_labeling.md §5-6) showed H5 "Approval-workflow
adherence" with 0 satisfied across all splits and 99.85%-100% not_applicable. Root cause:
in configs/policy_taxonomy.json, H5 has `maps_pku_categories: []` and no UF/soft mapping,
so no data source contains an H5-applicable scenario. The 32B teacher correctly returned
not_applicable; this is a DATA/DESIGN gap, not a labeling bug (H5's definition IS passed to
the teacher via build_messages, confirmed). A near-all-N/A policy cannot contribute
satisfied/violated contrast to the G2 FN-asymmetry analysis (P5).

Decision (human-approved direction, 2026-07-15): keep the 5-hard-policy structure; REPLACE
H5's concept with a hard-safety dimension that IS backed by the local data, chosen from a
PKU harm category that is (a) currently UNUSED by H1-H4 and (b) well-populated so both
satisfied and violated coverage are guaranteed after sampling. H1-H4, S1-S3, T1-T2 unchanged.

Why this over alternatives: self-building H5 data (option B) adds a synthetic dependency to
a HARD-safety policy that anchors the FN-asymmetry claim — undesirable to have the paper's
safety core rest on templated data. Dropping to a 9-policy stack (option C) weakens the hard
layer from 5 to 4. Reusing real PKU categories keeps the hard layer grounded in the same
distribution as H1-H4.

Procedure (in order):
1. Run scripts/day2/diagnose_pku_categories.py to get the ACTUAL harm-category distribution
   and which are already used by H1-H4. (No guessing — pick from real counts.)
2. PaperGuru picks the replacement category/definition for H5 from the diagnostic output.
3. Update configs/policy_taxonomy.json H5 (name/definition/maps_pku_categories) and, if the
   H5 short strings in src/policy_defs.py hardcode the old name, update them consistently.
4. Regenerate the pool and RE-LABEL train/calib/test (Day-2) with the same seed/schema; the
   pool membership will change because H5 mapping affects nothing in sampling BUT the labels
   must be regenerated so H5 reflects the new definition. Re-run the static audit and confirm
   H5 now has satisfied AND violated in all splits.

Impact on propositions/gates: P1-P6 definitions unchanged; H5's *concept* changes but its
ROLE (a medium-severity hard policy feeding FN-asymmetry) is preserved. All Day-2 labels are
regenerated, so the prior labels (checksums in day2_labeling.md §7) are SUPERSEDED. No
downstream (G1-G4) result exists yet, so nothing already-reported becomes provisional beyond
the Day-2 labels themselves.

Reversibility: revert policy_taxonomy.json H5 and re-label. Fully reversible; costs one
re-label pass (~9 min on 2 GPUs per Day-2 throughput).

Open question for PaperGuru: RESOLVED.

## Final decision (2026-07-15, after diagnostic e27dd24)
PKU category diagnosis (logs/pku_categories.log, full 73,907 rows / 147,814 responses)
listed these UNUSED, well-populated categories: White-Collar Crime (9,846),
Psychological Harm (9,243), Insulting Behavior (7,852), Endangering National Security
(6,438), Disrupting Public Order (4,819), Human Trafficking (4,689), Discriminatory
Behavior (3,711).

CHOSEN: **H5 = "National-security safeguard"** mapped to PKU category
**"Endangering National Security"** (6,438 harm-present responses). Rationale:
- Ample two-class coverage guaranteed (6,438 harm-present >> the previous 0).
- ORTHOGONAL to H1-H4: H1 privacy, H2 economic/cyber crime, H3 violence/physical/drugs,
  H4 mental manipulation. National security is a distinct high-risk semantic domain,
  avoiding the overlap risk of White-Collar Crime (~H2) or Psychological Harm (~H4).
- Clear refuse/comply boundary — ideal for the FN-asymmetry (P5) signal: "unsafe scored
  safe" is unambiguously dangerous here.
- Severity kept at MEDIUM to preserve H5's original role/weight in severity_weights;
  did not promote to high to avoid perturbing the severity distribution.

Sampling note: PKU sampling in src/sample_data.py is random/unbiased (not category-
targeted). At 6,438/147,814 ≈ 4.4% base rate, a 6,400-item PKU draw yields ≈ 280 H5-
applicable items — sparse relative to H1-H4 but well above the two-class coverage floor.
Kept sampling unbiased on purpose (targeted oversampling of one hard policy would distort
the per-policy comparison central to G1/P3). If the re-labeled audit shows H5 two-class
coverage is still marginal, revisit (would then be a new Yellow change).
